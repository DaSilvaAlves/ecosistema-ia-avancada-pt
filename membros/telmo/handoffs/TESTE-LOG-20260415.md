### Passo 4 — Clicar "Começar" em Chrome anónima — FALHA SILENCIOSA

- **Acção:** clicar "Começar" no passo 6 do onboarding (Chrome anónima)
- **Resultado:** **Nada acontece**. Sem banner vermelho, sem navegação, sem erro no Console.
- **Debug feito:**
  - Console completamente vazio após click (filtro "Default levels" todos marcados)
  - Issues do DevTools revelam 5 issues, uma delas crítica:
    - 🟠 **"Chrome may soon delete state for intermediate websites in a recent navigation chain"** — afecta `gitcblghucmpzizpxxki.supabase.co`
- **Causa raiz identificada:** **Chrome Bounce Tracking Mitigations** em aba anónima está a restringir state do domínio Supabase. O fluxo magic link redirecciona através de `supabase.co` sem "user interaction", então Chrome pode limpar cookies/storage silenciosamente. Efeito: sessão local parece existir mas tokens podem falhar ao fazer request → `updateProfile` não resolve nem rejeita → silent hang.
- **Bug arquivado:**
  - 🔴 **B6:** Auth flow falha silenciosamente em Chrome anónima por bounce tracking. Fix arquitectural: ou (a) mudar para PKCE flow, ou (b) aceitar limitação e avisar utilizadores que anónima pode não funcionar, ou (c) usar deep-link storage anti-tracking-friendly.

### Passo 7 — Investigação profunda pós-redeploy (15/04 ~03:00)

- **Sintoma:** Mesmo com `e163213` verde no Vercel, spinner eterno ao abrir https://fitcoach-ai-lake.vercel.app/ em Chrome normal, mesmo após hard refresh.
- **Root cause real identificada:** `useAuth.tsx` linhas 47-62 — callback de `onAuthStateChange` era `async` e fazia `await supabase.from('profiles').select(...)` DENTRO do callback. Isto é **anti-padrão documentado da Supabase v2**: o auth client fica locked durante o callback. Qualquer chamada Supabase que internamente precise refresh de token entra em deadlock. `setLoading(false)` nunca era chamado.
- **Fix correcto aplicado:** `b2fbaa7` — `fix(auth): move profile fetch out of onAuthStateChange callback to kill deadlock`.
  - Callback `onAuthStateChange` passou a síncrono. Só seta `user`.
  - `fetchProfile` movido para `useEffect` separado com `user?.id` como dependência.
  - `getSession()` também deixou de fazer fetchProfile.
  - Safety net: `setTimeout` de 10s força `setLoading(false)` se algo ainda travar.
- **Resultado:** Spinner eterno do boot/tab-switch **RESOLVIDO**. Dashboard aparece instantaneamente. ✅ **B6-code PASSED.**

### Passo 8 — Clicar "Gerar Novo Plano"

- **Acção:** no dashboard, clicar "Gerar Novo Plano".
- **Resultado:** **Dashboard desapareceu** — substituído por spinner fullscreen. Ficou preso indefinidamente.
- **Root causes identificadas (duas, empilhadas):**
  1. **UX bug** — `DashboardPage.tsx:58`: `const loading = workoutLoading || progressLoading; if (loading) return <Spinner />`. O `useWorkout.ts:97` fazia `setLoading(true)` durante `generatePlan()`, disparando o spinner fullscreen da página inteira.
  2. **Infra bug** — `vercel.json` não tinha configuração de `maxDuration`. Plano Hobby default = **10 segundos**. Claude Sonnet 4.5 a gerar plano demora 15-30 segundos. → **silent 504** → fetch resolvia com erro genérico → UI presa no spinner (por causa do bug #1).
- **Fix aplicado:** `d6ec516` — `fix(plan-gen): stop hiding dashboard during generatePlan + raise Vercel timeout to 60s`.
  - Removido `setLoading(true)/(false)` de `generatePlan()`. Caller já tem `generatingPlan` local state.
  - `vercel.json` → `"functions": { "api/generate-plan.ts": { "maxDuration": 60 } }`.
  - Adicionado `AbortController` com timeout de 65s no client para surfacar erro em PT-PT se realmente travar.
- **Resultado:** ✅ **"Gerar Novo Plano" FUNCIONA.** Eurico confirmou em https://fitcoach-ai-lake.vercel.app/.
