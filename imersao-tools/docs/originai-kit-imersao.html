P1
Briefing Generator
Transforma a tua dor num briefing estruturado em 2 minutos
briefing-generator.expressia.pt 5 min 2 ficheiros na pasta
O que fazer
  1. Cria uma pasta para o teu projecto e dentro dela uma pasta docs/
  2. Abre briefing-generator.expressia.pt
  3. Preenche os 4 passos:
    — Passo 1: Nome do projecto + nivel de experiencia
    — Passo 2: A tua dor (o problema que queres resolver)
    — Passo 3: 3 funcionalidades principais (uma por linha)
    — Passo 4: Para quem e a solucao + o que NAO queres (opcional)
  4. No final, a ferramenta gera dois blocos. Guarda cada um separadamente:
    — Clica em "Copiar Payload de Orquestracao" → abre o Bloco de Notas → cola → guarda como docs/payload.md
    — Clica em "Copiar Blueprint para Claude Code" → abre o Bloco de Notas → cola → guarda como docs/blueprint.md
Exemplo real -- Telmo (Personal Trainer Virtual)
Nome: Personal Trainer Virtual
Nivel: Avancado
Dor: "Pessoas terem um personal trainer virtual, porque nem todas conseguem pagar 200/300EUR por mes a um personal trainer"
Funcionalidades: planos de treino, acompanhamento, correccao de exercicios/nutricao
Para quem: pessoas que nao conseguem pagar 200/300EUR por mes a um personal trainer
O que recebes
Dois ficheiros na tua pasta:

1. docs/payload.md — Payload de Orquestracao: estrutura tecnica do projecto (scaffolding, stack, accoes)
2. docs/blueprint.md — Blueprint Estrategico: visao, posicionamento, directrizes de design e execucao

Juntos, sao o teu briefing base. Mas ainda e generico — a ferramenta nao sabe nada sobre o teu mercado.

PARA! Nao copies estes blocos para o terminal. Ainda nao. Falta o passo mais importante: PESQUISA.
Porque nao avancamos ja para o codigo
O Briefing Generator da-te um briefing generico. Nao sabe nada sobre o mercado de fitness virtual, apps concorrentes, precos praticados, modelos de negocio que funcionam. Se construires com isto, vais ter uma app tecnicamente correcta mas comercialmente cega.
P2
Prompt de Pesquisa
Pede a IA para criar um prompt de pesquisa optimizado para o TEU caso
Claude Code / Gemini CLI 5 min Prompt de pesquisa optimizado
O que fazer
  1. Abre o terminal na pasta do teu projecto (Claude Code ou Gemini CLI)
  2. Diz a IA que tens o teu briefing em docs/payload.md e docs/blueprint.md
  3. Pede-lhe para gerar um prompt de pesquisa de mercado optimizado para o teu caso
  4. O objectivo: criar um prompt que vais colar em 3 LLMs diferentes (Perplexity, Gemini, GPT)
  5. A IA le os teus ficheiros e gera um prompt personalizado -- muito melhor do que generico

Prompt para copiar

Le o meu briefing em docs/payload.md e docs/blueprint.md Com base no meu projecto, gera um PROMPT DE PESQUISA DE MERCADO optimizado que eu vou colar em 3 LLMs diferentes (Perplexity, Google Gemini e ChatGPT). O prompt deve pedir: 1. MERCADO -- Que solucoes ja existem? Concorrentes directos e indirectos? Precos praticados? 2. PUBLICO -- Quem sao estas pessoas? O que ja tentaram? Porque falharam? 3. MODELO DE NEGOCIO -- Como e que projectos semelhantes ganham dinheiro? 4. FUNCIONALIDADES -- Que features sao essenciais? O que diferencia os bons dos maus? 5. TECNOLOGIA -- Que APIs, servicos ou integracoes podem acelerar? Regras: - Perguntas especificas para o MEU projecto (nao genericas) - O prompt deve funcionar em qualquer LLM - No final do prompt, inclui esta instruccao: "Com base nesta pesquisa, cria um PRD tecnico para uma equipa de desenvolvimento de software. O PRD deve incluir: funcionalidades validadas, stack tecnica recomendada, modelo de negocio, e MVP com criterios de aceitacao."
Porque pedir a IA para gerar o prompt
A IA ja leu o teu briefing. Sabe o nome do projecto, a dor, as funcionalidades, o publico. O prompt que ela gera e personalizado -- em vez de "que apps existem?", gera "que apps de personal trainer virtual existem em Portugal e quanto cobram?". Muito mais eficaz.
O detalhe importante
Repara na ultima instruccao do prompt: pedes a cada LLM para criar um PRD tecnico alem de pesquisar. Isto e intencional -- a LLM que fez a pesquisa tem o contexto completo para transformar os dados num documento pronto para construcao.
P3
Pesquisar + PRD Tecnico
3 LLMs pesquisam o teu mercado e entregam-te um PRD pronto
3 LLMs a tua escolha 20-30 min 3 pesquisas + 3 PRDs tecnicos
O que fazer
  1. Abre 3 separadores com as 3 LLMs que mais usas (ex: Perplexity, ChatGPT, Grok, Google Gemini, Claude — escolhe as tuas)
  2. Em cada um, cola o prompt do Passo 2 (inteiro, de uma vez)
  3. Cada LLM vai fazer duas coisas: pesquisar o mercado e criar um PRD tecnico
  4. Para cada LLM, o processo e sempre o mesmo:
    — Copia o resultado completo
    — Abre o Bloco de Notas
    — Cola
    — Guardar como na pasta docs/ do teu projecto
  5. Nomes dos ficheiros (adapta ao nome da LLM que usaste):
    docs/pesquisa-1.md
    docs/pesquisa-2.md
    docs/pesquisa-3.md
Porque guardar em ficheiros e nao copiar para o terminal
Cada pesquisa pode ter milhares de palavras. Se colares 3 pesquisas directamente no terminal, vais estourar o contexto da IA e a qualidade cai. Ficheiros resolvem isto: a IA le cada um separadamente, com calma, sem perder informacao.
Porque 3 e nao 1
Cada LLM tem personalidade, dados e angulos diferentes. Usa as 3 que ja conheces e com quem ja trabalhas — elas ja tem contexto sobre ti.

3 fontes = triangulacao. Se as 3 dizem a mesma coisa, e verdade. Se divergem, encontraste uma oportunidade.
Porque as LLMs criam o PRD e nao o Claude Code
A LLM que fez a pesquisa tem o contexto completo -- sabe exactamente o que encontrou. Se passasses os dados crus para o Claude Code, ele estaria a interpretar em segunda mao. Assim, o PRD tecnico chega ja pronto e fundamentado nos dados reais.
P4
PRD Tecnico Final
Funde as 3 pesquisas num PRD unico pronto para construcao
Claude Code / Gemini CLI 5 min PRD final pronto para construcao
O que fazer
  1. Confirma que a tua pasta docs/ tem tudo:
    — payload.md + blueprint.md (Passo 1)
    — pesquisa-perplexity.md + pesquisa-chatgpt.md + pesquisa-gemini.md (Passo 3)
  2. Abre o terminal na pasta do teu projecto (Claude Code ou Gemini CLI)
  3. Cola o prompt abaixo -- a IA le tudo e funde num PRD unico

A tua pasta neste momento

meu-projecto/ └── docs/ ├── payload.md ← Passo 1 ├── blueprint.md ← Passo 1 ├── pesquisa-1.md ← Passo 3 (LLM 1) ├── pesquisa-2.md ← Passo 3 (LLM 2) └── pesquisa-3.md ← Passo 3 (LLM 3)

Prompt para copiar

Le o meu briefing base em docs/payload.md e docs/blueprint.md e as 3 pesquisas de mercado em docs/pesquisa-1.md, docs/pesquisa-2.md e docs/pesquisa-3.md Cada pesquisa ja inclui um PRD tecnico gerado pela LLM que pesquisou. Cruza os 3 PRDs com o briefing base e funde tudo num PRD FINAL unico. O PRD final deve: 1. Usar as funcionalidades VALIDADAS pelas pesquisas (nao as genericas do briefing) 2. Incluir o modelo de negocio recomendado (o que aparece em pelo menos 2 das 3 fontes) 3. Usar as APIs e servicos concretos identificados 4. Definir o MVP com criterios de aceitacao concretos 5. Se as fontes divergem, indicar e escolher a opcao mais fundamentada Formato: - Visao do projecto (2-3 frases) - Problema + publico (com dados das pesquisas) - MVP: funcionalidades com criterios de aceitacao - Stack tecnica com justificacao - Modelo de negocio - Metricas de sucesso Grava o resultado em docs/prd-final.md Este PRD vai ser entregue directamente para construcao. Deve ser claro, concreto e sem ambiguidade.
A diferenca
Briefing generico (Passo 1): "Maximizar a percepcao de valor para o utilizador final. Clean Code e padroes DRY."

PRD final (Passo 4): "App freemium de treino personalizado com IA. Plano gratuito: 3 treinos/semana gerados por IA com base no perfil. Plano Pro (8EUR/mes): treinos ilimitados + nutricao + progresso. API ExerciseDB para base de exercicios. Diferenciador: planos adaptados ao equipamento que o utilizador TEM (casa vs. ginasio)."

A diferenca? Pesquisa. O primeiro e generico. O segundo e fundamentado em dados reais de 3 fontes diferentes.

L A Narrativa LEGO — De Tarefas a Sistemas

A analogia central do Bloco 6 (Tarefas + Tiers). Mostra como pecas simples constroem sistemas complexos — exactamente como tarefas bem definidas constroem projectos.

A Progressao: Do Caos ao Resultado

1Caos
→
2Organizar

4Instrucoes
→

1. Caos — Pecas soltas sem organizacao. E assim que comecam a maioria dos projectos: ideias espalhadas sem estrutura.
2. Organizar — O processo de separar e categorizar. E o que fazes no Brief e no Detalhamento.
3. Agrupar — Pecas agrupadas por tipo. Cada saco e uma Etapa — contem tudo o que precisas para aquela fase.
4. Instrucoes — Passos 1-9 claros e sequenciais. Cada passo e uma Tarefa: verbo + resultado verificavel.
5. Construir — Execucao com guia. Sabes exactamente o que fazer a seguir. Sem duvidas, sem improvisos.
6. Resultado — Projecto completo. Todas as etapas executadas, todas as tarefas verificadas.
7. MVP — Versao simples. Mesma logica, menos pecas. Comeca pequeno, valida, depois expande.
A licao: Nao e sobre o tamanho do projecto. E sobre a qualidade da decomposicao. Pecas simples, bem organizadas, constroem qualquer coisa.

Escala: Blocos Simples → Sistemas Complexos

Cidade LEGO — Cada loja, restaurante, rua — construida com as mesmas pecas basicas. Sistemas complexos de blocos simples.
McLaren F1 — Engenharia de precisao. Milhares de pecas simples → resultado excepcional.

Regras de Quebra

Etapa vs Tarefa — Etapa: objectivo intermediario, 3-7 tarefas. Tarefa: accao especifica, verbo, verificavel.
Fluxo de Quebra — "O que quero alcancar?" → Etapas → Tarefas → Teste de qualidade.

Analogia da Casa (Bloco 4)

Construir uma Casa — Fundacao (Cavar, Nivelar, Concretar) → Paredes (Levantar, Rebocar, Instalar) → Telhado (Estrutura, Telhas, Calhas)
A Planta — O Brief e a tua planta tecnica. Sem planta, constróis a casa da imagem 18.

Sem Projecto vs Com Projecto

Sem projecto — "Isto e o que temos." Construcao sem planta, sem metodo, sem clareza.
Com projecto — "Isto e o que construiste." O resultado de clareza + metodo + execucao.

B Blueprint do Arquitecto Front-End 2025

Os 7 Pilares tecnicos para o Bloco 7 (Pipeline Tecnico). O metodo que transforma codigo caotico em arquitectura profissional.

O Blueprint — Target: Next.js/React | Engine: Tailwind v4 | Strictness: Maximum

Zona Humana vs Zona Maquina

A Regra de Ouro: Esgotar o cerebro humano no papel primeiro. O briefing final aprovado dita a operacao da IA em minutos, nao em dias.

As 5 Fases da Auditoria

Descoberta → Planeamento → Auditoria → Recomendacoes → Plano de Accao

Os 7 Pilares

PILAR 1
Arquitectura React/Next.js
Component Tree Topology. Minimizar 'use client'. Server Components por definicao. Extraccao de Hooks. Keys estaveis. Estado derivado em vez de useEffect.
PILAR 2
Tailwind CSS (v3 → v4)
Oxide Engine (Rust): Full build 378ms → 100ms. CSS-first config com @theme. OKLCH. Container Queries built-in. Breaking changes: shadow→shadow-sm, outline-none→outline-hidden.
PILAR 3
Design Tokens (W3C DTCG)
3 camadas: Global/Primitive (blue-500) → Semantic (color-primary) → Component (button-primary-bg). OKLCH para percepcao uniforme. JSON DTCG v2025.10.
PILAR 4
Ecossistemas de UI
shadcn/ui (~2KB) vs Chakra UI (~45KB) vs Material-UI (~87KB). Filosofia copy-paste ownership: zero vendor lock-in, tree-shaking perfeito. Tu manténs o codigo.
PILAR 5
Acessibilidade (WCAG 2.2 + APCA)
Contraste 4.5:1 texto, 3:1 UI. focus-visible obrigatorio. sr-only em botoes com icone. APCA alinha com OKLCH. motion-reduce respeitado.
PILAR 6
Performance
